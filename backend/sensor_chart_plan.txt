================================================================================
SYSTEM SENSOR CHARTS — IMPLEMENTATION PLAN
================================================================================


1. OBJECTIVE
--------------------------------------------------------------------------------

Build a chart per discovered system (SYS_1, SYS_2, etc.) displaying:

  - Raw sensor value traces (one line per sensor in that system)
  - Downtime bands (shaded regions where the machine is off)
  - Alert episode flags (shaded regions where anomalies were detected)


2. WHAT ALREADY EXISTS IN THE PIPELINE
--------------------------------------------------------------------------------

The following files are already produced by the pipeline and contain the
data needed for downtime bands, alert overlays, and system membership.

  dynamic_catalog.csv
    - Created in Cell 8 (Dynamic System Discovery)
    - Contains the mapping of which sensors belong to which system
    - Columns: system, sensor
    - Example: SYS_1 → DESF_TA__KW_TOT, DESF_TA__J1_I_AVG, ...

  scores.csv
    - Created in Cell 18 (Assemble + Save)
    - Contains one row per minute-timestamp
    - The "mode" column has "RUNNING" or "DOWNTIME" per timestamp
      → This is used to derive downtime bands
    - Also contains system-level anomaly scores (score_SYS_1, score_SYS_2, ...)
      and risk_score, class, sqs_mean, physics_score, subsystem_score

  alerts.csv
    - Created in Cell 17 (Alert Episodes)
    - Contains alert episodes with start_ts, end_ts, duration, severity, class
    - The "class" column indicates which system triggered the alert
      (e.g. SYS_1, SYS_2, PROCESS, INSTRUMENT)
    - The "severity" column is HIGH or MEDIUM
      → This is used to overlay alert bands on the chart

  alerts_sensor_level.csv
    - Created in Cell 17
    - Contains per-sensor detail within each alert episode
    - Columns: start_ts, end_ts, sensor, sensor_rank, sensor_peak_score
    - Useful for highlighting which specific sensors drove an alert

  sensor_config.csv
    - Created in Cell 9 (Learn Bounds)
    - Contains learned bounds per sensor (p0_1, p99_9, roc_p99_9)
    - Useful for showing normal operating ranges on charts

  risk_sensor_decomposition.csv
    - Created in Cell 16 (Block Scoring)
    - Contains per-sensor risk contribution at each timestamp
    - Useful for detailed drill-down but not required for the main chart

  dynamic_weights.csv
    - Created in Cell 18
    - Contains fusion and risk weights per system
    - Useful for displaying system importance metadata


3. WHAT IS MISSING
--------------------------------------------------------------------------------

The pipeline processes raw sensor data into a wide DataFrame called "df"
(one column per sensor, one row per minute) but does NOT save this DataFrame.
Only derived scores and alerts are saved.

  Missing file: Raw sensor values per system

    The wide DataFrame "df" exists in memory during Cell 4 through Cell 18
    but is never written to disk. This DataFrame contains the actual sensor
    readings (voltage, current, temperature, etc.) that need to be plotted
    as line traces on the chart.


4. WHAT NEEDS TO BE ADDED TO THE PIPELINE
--------------------------------------------------------------------------------

A new save block in Cell 18 that writes the raw sensor values from "df"
grouped by system membership from "catalog".

  Two approaches:

    Per-system CSV files (recommended)
      One file per system: sensor_values_SYS_1.csv, sensor_values_SYS_2.csv, etc.
      Each file has timestamp_utc as the index and one column per sensor in
      that system. Keeps individual file sizes manageable.

    Single long-format CSV
      One file (sensor_values_long.csv) with columns:
      timestamp_utc, sensor_id, system, value
      Easier to serve from a single API endpoint but potentially very large.

  The per-system approach is recommended because each API request only needs
  to load one system's data at a time.


5. WHAT NEEDS TO BE ADDED TO THE BACKEND
--------------------------------------------------------------------------------

  Systems list endpoint
    Returns the list of discovered systems with their sensor counts and
    sensor names. Sources from dynamic_catalog.csv.

  System sensor values endpoint
    For a given system_id, returns:
      - The raw sensor time series (from the new sensor_values_SYS_*.csv)
      - Downtime bands derived from scores.csv mode column
      - Alert episodes filtered to that system from alerts.csv
    Should support a downsample parameter to control how many points are
    returned (every Nth point) for frontend performance.

  System scores endpoint (optional)
    Returns the system-level anomaly score time series (score_SYS_* column
    from scores.csv) for overlaying the anomaly score as a secondary axis
    on the sensor chart.


6. WHAT NEEDS TO BE ADDED TO THE FRONTEND
--------------------------------------------------------------------------------

  API helper functions
    Functions to call the new backend endpoints for systems list and
    system sensor data.

  System charts page or section
    A container that loads the systems list and renders one chart per system
    (or allows selecting a system via tabs/dropdown).

  Individual system chart component
    Uses a charting library (Recharts recommended since it is likely already
    in the project) to render:
      - Line traces for each sensor in the system
      - Gray shaded regions for downtime bands (using ReferenceArea)
      - Red/orange shaded regions for alert episodes (using ReferenceArea)
      - Multiple y-axes if sensors have very different scales

  System selector
    Tabs, dropdown, or accordion to choose which system chart to display.
    Avoids loading all systems at once for performance.

  Downsample control
    A slider or dropdown to let the user control data resolution.
    Default of every 5th point (about 2000 points per day) keeps the
    chart responsive.


7. DATA FLOW
--------------------------------------------------------------------------------

  Pipeline produces:
    dynamic_catalog.csv       → system membership
    sensor_values_SYS_*.csv   → raw sensor readings (NEW)
    scores.csv                → mode column for downtime bands
    alerts.csv                → alert episodes for overlay

  Backend combines:
    Reads catalog to know which sensors belong to which system
    Reads sensor values CSV for the requested system
    Derives downtime bands from scores.csv mode column
    Filters alerts.csv to the requested system
    Returns everything in one JSON response

  Frontend renders:
    Line chart with one trace per sensor
    Gray ReferenceArea bands for each downtime interval
    Red/orange ReferenceArea bands for each alert episode
    Optional secondary y-axis for the system anomaly score


8. FILE INVENTORY AFTER IMPLEMENTATION
--------------------------------------------------------------------------------

  Existing files (no changes needed):
    scores.csv
    alerts.csv
    alerts_sensor_level.csv
    sensor_config.csv
    risk_sensor_decomposition.csv
    dynamic_catalog.csv
    dynamic_weights.csv

  New files (from pipeline addition):
    sensor_values_SYS_1.csv
    sensor_values_SYS_2.csv
    sensor_values_SYS_N.csv    (one per discovered system)
    sensor_values_ISOLATED.csv (optional, for unclustered sensors)


9. PERFORMANCE NOTES
--------------------------------------------------------------------------------

  The raw sensor data can be large (8640 rows per day × N sensors per system).
  For a 7-day dataset this is about 60,000 rows per system.

  The backend endpoint should support downsampling (return every Nth row)
  with a default of 5, reducing to about 12,000 points per system.

  The frontend should lazy-load system data only when the user selects
  that system, rather than fetching all systems on page load.

  The per-system CSV approach means each API call only reads one small file
  rather than scanning a single large file.


================================================================================
END OF PLAN
================================================================================
